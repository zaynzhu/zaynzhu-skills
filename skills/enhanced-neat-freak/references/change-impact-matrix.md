# 变更影响矩阵

遇到"这条事实应该写到哪里"时，先查本矩阵。目标是让每条知识只出现在最适合的层级，避免重复和过期。

## 读者分层

| 知识层 | 主要读者 | 应写内容 | 不应写内容 |
|--------|----------|----------|------------|
| README | 第一次接触项目的人 | 项目定位、安装、快速开始、常用入口 | 内部机制全文、历史流水账 |
| docs | 使用者、维护者、下游系统 | 接入、架构、运维、排障、交接 | agent 私有偏好 |
| CLAUDE.md / AGENTS.md | 下次写代码的 agent | 边界、禁区、命令、项目约定、验证方式 | 变更日志、长篇架构说明 |
| memory | 跨会话 agent | 长期偏好、稳定坑点、跨项目事实 | 单次任务记录、已完成待办 |
| CHANGELOG / handoff | 交接读者 | 阶段完成情况、迁移说明、风险 | 永久开发规则 |

## 代码变更到文档层

| 变更类型 | README | docs | CLAUDE.md / AGENTS.md | memory |
|----------|--------|------|------------------------|--------|
| 新增命令行入口 | 快速开始或命令表 | 详细参数和示例 | 常用验证命令 | 通常不写 |
| 新增 API / 路由 | 简短入口或链接 | integration、architecture、错误码 | 路由边界和测试命令 | 仅跨项目稳定协议才写 |
| 新增环境变量 | 必填配置摘要 | runbook、部署、下游接入 | 环境变量表和安全注意 | 仅长期跨项目约定才写 |
| 新增数据库表 | 通常不写 | architecture data model | 表名和边界规则 | 通常不写 |
| 重命名模块/目录 | 若用户会看到则写 | 更新相关引用 | 项目结构表 | 若下次易踩坑可写 |
| 删除功能 | README 删除入口 | docs 标记迁移或删除 | 删除旧规则 | 删除旧记忆 |
| 修复 bug | 通常不写 | 若影响运维则写排障 | 若是高复发坑点，写规则 | 只写可复用坑点 |
| 重构内部实现 | 通常不写 | 架构图或模块边界变化才写 | 边界变化才写 | 通常不写 |
| 发布/上线 | README 不写历史 | CHANGELOG、handoff | 不写历史叙事 | 通常不写 |

## 反向清理矩阵

| 发现的问题 | 优先动作 | 判断依据 |
|------------|----------|----------|
| 同一命令在三处不同 | 选一个权威位置，其余改链接 | 读者只需要一个可信入口 |
| `CLAUDE.md` 顶部有历史 blockquote | 删除或迁到 CHANGELOG | 规则文件不承载历史 |
| README 写了内部状态机 | 摘要后迁到 architecture | README 只负责入口 |
| docs 有"今天/最近/上周" | 改成绝对日期或删除 | 文档会跨时间阅读 |
| memory 记录已完成待办 | 删除 | 记忆不是任务日志 |
| 旧路径仍在文档里 | 全仓库搜索后替换或标注迁移 | 避免读者复制错误路径 |

## 跨项目影响

如果本次变更涉及以下任一项，检查下游项目文档：

- 共享 API、SDK、CLI、配置文件或协议。
- 共享域名、路由、环境变量或部署流程。
- 上游服务的鉴权、限流、错误码或数据格式。
- 生成给其他项目消费的文件或包。

跨项目处理原则：

1. 当前项目内的权威文档先更新。
2. 下游项目能确定路径且改动小，则同步更新。
3. 下游范围不明时，在交付的风险项里列出待检查项目和关键词。

## 信息应该进入记忆的判断

只有满足下面三项才建议写入记忆：

1. 下次会话很可能复用。
2. 不写会导致错误假设或明显返工。
3. 不能只靠 README/docs 快速发现。

反例：

- "今天修了 README"：不写。
- "某次测试失败是因为临时网络问题"：不写。
- "这个用户要求所有 commit 用中文 type 前缀"：可写。
- "这个仓库新增 skill 必须同步 README 和 CLAUDE.md"：可写到项目规则，未必写记忆。
