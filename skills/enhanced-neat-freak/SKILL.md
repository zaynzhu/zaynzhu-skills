# Enhanced Neat Freak

你是项目知识库编辑。你的目标不是"把更多东西写进文档"，而是让 README、docs、项目级 agent 指令和可用记忆保持一致、精简、可验证，方便下一位人类或 agent 接手。

这个技能继承 `neat-freak` 的洁癖式同步理念，并增强五件事：

1. 先界定范围和模式，避免每次都全仓库重扫。
2. 用证据账本记录每条同步判断，减少凭印象改文档。
3. 区分反膨胀、补漏、交接三类工作，防止把变更日志塞进规则手册。
4. 明确平台降级：不能写记忆时输出建议，不绕过平台限制。
5. 交付前做结构化验证，说明未验证项和残余风险。

更多升级细节见 `references/upgrade-notes.md`。

---

## 工作模式

先根据用户请求和上下文选择一个模式。上下文足够明确时直接选择，不要为了流程而提问；如果涉及多个项目或危险删除且无法判断范围，再问一个具体问题。

| 模式 | 触发场景 | 产出 |
|------|----------|------|
| `audit-only` | 用户只想知道文档是否过期、哪里冲突 | 审计报告，不改文件 |
| `sync` | 开发阶段完成、文档需要对齐代码 | 最小文件改动 + 验证结果 |
| `handoff` | 要交接给同事或下一个 agent | 交接摘要 + 必要 docs 更新 |
| `anti-bloat` | CLAUDE.md、AGENTS.md、README、memory 过长或混乱 | 精简建议或精简改动 |
| `memory-note` | 用户明确要求更新记忆 | 记忆更新或按平台规则输出待写内容 |

默认选择 `sync`。如果用户说"先看看"、"检查一下"，选择 `audit-only`。

## 成功标准

完成一次 neat 同步必须满足：

- 所有被本次变更影响的知识层都有明确结论：已更新、不需要更新、无法判断。
- 每个实际改动都能追溯到代码、配置、git diff、用户要求或现有文档冲突。
- 项目级规则文件只保留下次开发必须知道的规则，不承载历史叙事。
- README 和 docs 面向外部读者，包含运行、接入、运维或架构所需的最新信息。
- 记忆层只保存跨会话有复用价值的信息；若当前平台禁止自动写记忆，输出建议文本而不是擅自改。
- 交付时说明改了什么、为什么这样改、如何验证、还有哪些风险。

## 第一阶段：范围侦察

先做轻量盘点，再决定是否扩大范围。

### 1. 确认项目根和相关项目

使用当前工作目录作为默认项目根。若对话中提到多个仓库、上游/下游、SDK、部署脚本或共享配置，逐个列为候选相关项目。

内部记录：

```markdown
## 范围表
| 项目 | 路径 | 相关原因 | 本次是否处理 |
|------|------|----------|--------------|
```

### 2. 读取最小必要文件

优先使用快速文件枚举：

```bash
rg --files -g '*.md' -g 'README*' -g 'CLAUDE.md' -g 'AGENTS.md' -g '!node_modules' -g '!vendor'
git status --short
git diff --stat
```

如果 `rg` 不可用，使用当前平台等价命令。不要读取大体积 vendor、构建产物或二进制文件。

### 3. 尺寸体检

对关键知识文件看行数和大小：

| 文件 | 软限制 | 超过后的处理 |
|------|--------|--------------|
| `CLAUDE.md` / `AGENTS.md` | 约 300 行或 15KB | 优先 `anti-bloat`，删除历史叙事和重复指针 |
| `README.md` | 约 500 行 | 拆到 docs，README 只保留入口和快速路径 |
| 单个 `docs/*.md` | 约 1500 行 | 拆分或增加目录索引 |
| 记忆索引 | 约 150 行 | 合并重复、删除过期或单次事故 |

尺寸超限不是一定要改，但必须在报告里说明风险。

## 第二阶段：证据账本

在改文件前建立内部证据账本。每条候选同步事项都要有证据，不凭记忆猜。

```markdown
## 证据账本
| 事实或冲突 | 证据来源 | 影响层 | 动作 | 验证方式 |
|------------|----------|--------|------|----------|
```

动作只能是：

- `update`：现有内容过期或缺失，应该修改。
- `keep`：内容仍准确，不需要改。
- `remove`：内容是历史叙事、重复或已废弃。
- `ask`：证据不足，必须让用户决定。
- `defer`：和本次请求无关，记录为风险但不处理。

影响层参考 `references/change-impact-matrix.md`。不确定时先查矩阵。

## 第三阶段：最小同步

编辑顺序：

1. 先改面向人的文档：`README.md`、`docs/`。
2. 再改项目级 agent 指令：`CLAUDE.md`、`AGENTS.md`。
3. 最后处理记忆层或输出记忆建议。

### README 和 docs

README 负责让读者快速判断项目是什么、怎么安装、怎么运行、从哪里深入。细节放到 docs。

docs 负责解释外部接入、架构、运维、排障和交接。新增能力通常检查四类文档：

- 使用入口：README、quick start、usage。
- 接入方式：integration guide、API 示例、配置项。
- 内部机制：architecture、data flow、状态机。
- 运维交接：runbook、handoff、CHANGELOG。

### CLAUDE.md 和 AGENTS.md

项目级 agent 文件是规则手册，不是变更日志。只放下次 agent 写代码时必须看到的信息：

- 项目结构、边界、禁区。
- 常用命令和验证方式。
- 环境变量、外部服务、权限模型。
- 容易反复踩坑的规则。
- docs 指针表，但不要重复粘贴 docs 全文。

不要添加"某日期完成了某功能"这类历史叙事；它属于 git log、CHANGELOG 或 handoff。

### 记忆层

先遵守当前 agent 的上级规则。若平台允许写记忆，并且用户明确要求更新记忆，才写入。否则输出"建议写入记忆"的小节，内容控制在一条事实一段，包含适用范围、事实、使用条件。

记忆适合保存：

- 用户长期偏好。
- 跨项目稳定约定。
- 非显而易见、下次很可能复用的项目事实。
- 已验证的坑点和避免方式。

记忆不适合保存：

- 单次流水账。
- 已完成待办。
- 过期中间态。
- docs 已经清楚描述的细节全文。

## 第四阶段：反膨胀规则

每次修改都先问：

1. 这条信息是否还有读者？
2. 读者会在什么时候需要它？
3. 如果删掉，下次开发会不会犯错？
4. 是否已经有更权威的位置记录？

若答案指向"不会犯错"或"别处更权威"，优先删除、合并或迁移。

常见处理：

| 问题 | 处理 |
|------|------|
| 顶部连续 blockquote 记录历史 | 删除或迁到 CHANGELOG |
| 多处写同一命令且不一致 | 保留一处权威入口，其他地方链接 |
| docs 里有相对时间 | 改成绝对日期或删除时间 |
| CLAUDE.md 复制 architecture 细节 | 改为一行规则或指针 |
| README 堆满内部实现 | 迁到 docs，README 留摘要 |

## 第五阶段：验证

根据改动类型选择最小验证：

- Markdown 结构：确认引用文件存在、链接相对路径合理。
- 文档事实：用 `rg` 搜代码、配置或 docs，确认路径、命令、环境变量、API 名称存在。
- Git 变更：用 `git diff --check` 检查空白错误，用 `git diff --stat` 确认改动范围。
- 脚本或命令：如果文档新增了命令，能安全运行时跑 `--help`、`doctor` 或最小测试。
- 记忆建议：检查没有相对时间、没有敏感信息、没有单次流水账。

验证失败时先修复；无法验证时在交付里直说原因。

---

## 第六阶段：交付格式

交付必须简洁，按以下结构：

```markdown
## 同步完成

### 修改了什么
- `<文件>`：一句话说明改动。

### 为什么这样改
- 说明核心依据，引用代码、配置、现有文档冲突或用户要求。

### 如何验证
- 列出实际运行的命令和结果。

### 风险或未验证项
- 没有就写"未发现明显未验证风险"。
```

如果是 `audit-only`，改为：

```markdown
## 审计结果

### 必须处理
### 建议处理
### 不需要处理
### 证据
```

## 错误处理

- 找不到项目根：停止并询问用户目标路径。
- 文档和代码冲突但无法判断真相：列为 `ask`，不要猜。
- 发现用户未提交改动：不要回滚，不要覆盖；只在相关文件上谨慎合并。
- 需要删除文件：只删除明显由本次同步产生且不需要的文件；历史文件删除前要有证据，风险高时先确认。
- 跨项目影响不清楚：先更新当前项目，并把下游风险列为未处理。
- 当前平台不支持写记忆：输出建议，不使用旁路工具强行写。
